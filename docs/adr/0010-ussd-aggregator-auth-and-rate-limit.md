# ADR 0010 - USSD channel security: aggregator shared secret + in-process rate limit

- **Status:** Accepted (2026-07-08). Hardens the USSD channel from [ADR 0006](./0006-ussd-qr-primary-channel.md) (USSD/QR as the primary customer channel). This is the "Gate A" transport hardening in [`docs/security-roadmap.md`](../security-roadmap.md).
- **Context:** `/ussd` is a public HTTP endpoint that initiates real payment prompts. Two forces meet here. First, only the **contracted USSD aggregator** should be able to drive the channel in production; an open endpoint lets anyone push payment prompts. Second, even a legitimate path can be abused to **spray prompts at arbitrary DRC numbers** (a harassment / social-engineering vector, even though no money moves without the payer's operator PIN). The channel is a **thin caller** into the shared domain services (no money logic lives here, per the channel-agnostic rule), so this hardening is purely transport-level and belongs at the `/ussd` boundary. It must also not break the console's browser-driven dial simulator, which drives `/ussd` directly in local/sandbox.
- **Decision:**
  - **Aggregator authentication - a shared secret.** When `DRCPAY_USSD_SHARED_SECRET` is set, every request must carry it in the `X-USSD-Secret` header, compared in constant time (on encoded bytes, so a non-ASCII header is a clean 401, never a 500). The gate is **enabled by presence of the secret**: blank in local/sandbox so the dial simulator works, and **production refuses to boot without it** (fail-closed, alongside the sandbox-password and known-environment boot guards in `create_app`).
  - **Rate limiting - in-process, per customer number.** A per-msisdn sliding window (default 15/min) braked at the `/ussd` boundary. The msisdn is canonicalized to digits-only, so `+243...` and `243...` are one identity (one bucket, and one idempotency identity downstream). The window map self-evicts stale keys so an attacker spraying unique numbers cannot grow it without bound.
  - **Rejections respect the channel contract.** A rate-limited customer is in a live USSD session, so the throttle answers with a wire-format `END` message they can actually read. An auth failure is an aggregator/infrastructure error a correct integration never triggers, so it stays a plain `401`.
- **Consequences:**
  - In production only the contracted aggregator can drive the channel, and a missing secret or an unrecognized environment **fails fast at boot** rather than silently running open.
  - The in-process limiter is honest for **today's single-container deploy**; it does not coordinate across instances. A shared-state (e.g. Redis) limiter is deferred until we run more than one instance.
  - The shared secret is **symmetric** (both sides hold it); rotation is a coordinated env-var change with the aggregator. It authenticates the caller, not each request's integrity.
  - Rate limiting counts **requests, not completed sessions** - simple, and set above the worst-case legal session so honest customers are never cut off; a per-session limit would be tighter and is noted as future work.
- **Alternatives considered:**
  - **Mutual TLS (client certs)** - stronger caller authentication, but heavier to provision and rotate with the aggregator; deferred, not precluded.
  - **IP allowlist** - brittle (aggregator egress IPs change) and insufficient alone; usable later as defense-in-depth, not as the primary gate.
  - **Per-request HMAC / RFC-9421 signatures** (as we require on **inbound pawaPay callbacks**) - more robust than a static secret, but needs the aggregator to sign to our scheme; the aggregators we target send a static header, so a shared secret matches the integration reality. Revisit if an aggregator supports signing.
  - **Shared-state rate limiter (Redis)** - correct for multi-instance, unnecessary complexity for one container today.

# ADR 0009 — On-net payments: facilitate & record, not route

- **Status:** Accepted (2026-06-22). Builds on [ADR 0003](./0003-pass-through-no-custody.md) (pass-through, no custody). **Retires** the operator-API on-net "direct-collect" approach from the on-net foundation.
- **Context:** We tried to make same-network ("on-net") payments cheaper than pawaPay's two-leg collect+payout by having *our* system initiate a collection through the operator's API (a `DirectCollectRail`). Validating against Airtel's live developer portal + Collection API (v2.0) settled it: the collection request has **no payee / sub-merchant field** — it always credits **our** app's own merchant code. So the money would land in *our* wallet, needing a payout leg to reach the real merchant (no cheaper than pawaPay) and making us a **custodian** — which breaks [ADR 0003](./0003-pass-through-no-custody.md) and triggers the DRC **EMI licence** (~$2.5M capital). The capability we'd actually need — one credential collecting into many separate merchant accounts — is a partner-gated aggregator product the standard API doesn't expose.
- **Decision:** For same-network payments, **we do not move or hold the money.** The customer pays the merchant **directly on the operator's own rail**; our system **facilitates** (shows the customer how to pay) and **records & confirms** the sale for the merchant's books. Cross-network payments stay on **pawaPay** — the licensed custodian that keeps *us* non-custodial.
  - **Scope — all same-network pairs** (Airtel→Airtel, Vodacom→Vodacom, Orange→Orange). Facilitation needs no operator API, so it works for every operator.
  - **Fee = 0** on on-net — we never touch the money. Any future monetisation is a SaaS fee, never a per-transaction rail cut.
  - **Pay instruction — use the merchant's operator till ("buy goods") whenever they have one** (cheaper for the customer, shows the business name, and is the path that can later notify us automatically); **send-to-number (P2P) is the fallback**, used only for merchants without a till. Both are merchant-direct.
  - **Confirmation — merchant-attested.** MVP: the merchant taps **"Confirm received"** in the console after seeing the operator's SMS. Target: **auto-confirmation** via the operator's merchant-payment notification (no tap) once a merchant has a till + registered callback. Manual is the stopgap, not the destination.
  - **Provenance** — on-net "paid" is tagged **merchant-attested** in the ledger, visibly distinct from pawaPay's **rail-verified**. The books never overstate assurance.
- **Consequences:**
  - **Upholds ADR 0003** (non-custodial), needs **no operator money-API integration**, no EMI exposure, and is the **cheapest** path for same-network.
  - **Lower assurance** for on-net (merchant-attested) until operator notifications are wired — represented honestly, not hidden.
  - **Retire** the operator-API on-net machinery: `DirectCollectRail` + `request_direct_collection` (`ports.py`), `integrations/airtel/rail.py` + `integrations/mpesa/rail.py`, `integrations/simulated_direct.py`, the `container.direct_rails` wiring, the `DRCPAY_ONNET_SIMULATE` toggle, and the planned `DRCPAY_AIRTEL_*` credentials.
  - **Keep** `OnNetOrchestrator` (one customer→merchant ledger posting, `fee=0`, paid on an async confirmation — now triggered by the merchant/operator confirmation, not an operator callback), `use_on_net`, the `Charge` model, and the confirmation-endpoint seam.
  - **Add** the facilitation UX (the customer "pay them directly on <operator>" hand-off), a merchant **"Confirm received"** action, an optional **merchant till / merchant-code** field per network on the `Merchant`, and a **provenance** tag on the transaction.
  - The customer on-net experience changes from an in-app routed pay to a hand-off into the operator's own flow.
- **Alternatives considered:**
  - **Operator Collection API with us as the merchant** — rejected: custody + EMI exposure, and with no payee field it isn't even merchant-direct.
  - **Per-merchant operator credentials** — rejected: pushes an operator contract onto every merchant, undermines the aggregator value prop, heavy onboarding.
  - **Route on-net through pawaPay too** (status quo) — viable and rail-verified, but pays the full two-leg cost for same-network and misses the saving. Kept only as the cross-network path.

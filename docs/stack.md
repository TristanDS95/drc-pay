# DRC Pay — the full stack, in plain language

A tour of every piece of technology this app is built from, top to bottom: what each part is,
what it does for us, and why we chose it. No jargon assumed.

## The one-paragraph version
DRC Pay is a **website** (no app to install) with two faces: a **Merchant Console** for shop
owners and a **Customer page** a payer opens by scanning a QR code. Behind them sits a **Python
service** that holds all the money logic and talks to **pawaPay** — the company whose "rails" we
rent to actually move mobile money (Vodacom M-Pesa, Airtel, Orange). Everything runs as a single
**Docker** package hosted on **Railway**, with a **PostgreSQL** database keeping the records. We
never touch or hold the money ourselves — we just orchestrate it and take a small fee.

## Layer by layer

### 1. What people see — the web front-ends
- **Plain HTML, CSS, and JavaScript.** Two simple web pages, each a single file, with **no
  framework** (no React, no build step). You can open the file and read it top to bottom.
  - **Merchant Console** (`tooling/merchant-console/`) — the shop owner's cockpit: shows their QR
    code, takes payments, lists a live feed of transactions, and has a "Run reconciliation" safety
    button. It's behind a password.
  - **Customer page** (`tooling/customer-app/`) — what a payer sees after scanning the QR: enter an
    amount, pay, and watch it confirm in real time. No password, no login — a customer has no account.
- **Why so simple?** A payments demo shouldn't depend on a heavy front-end toolchain. Plain pages
  load instantly, are trivial to host, and are easy to reason about. (A polished native phone app is
  deliberately deferred until later.)
- **QR codes** are generated by a small library called **segno**.

### 2. The brain — the backend service
- **Language: Python** (version 3.12+). The whole money engine and web API are written in Python.
- **Web framework: FastAPI.** This is what turns Python functions into a web API the front-ends can
  call (e.g. "take this payment", "what's the status of this transaction"). It's fast, modern, and
  self-documents its endpoints.
- **Served by: Uvicorn.** The lightweight web server that actually runs FastAPI and answers requests.
- **Data validation: Pydantic.** Checks that every request coming in and response going out has the
  right shape (e.g. an amount is really a number) — a guardrail against bad data near real money.
- **How it's organized (hexagonal architecture):** the pure money logic lives in the middle and knows
  nothing about the web, the database, or pawaPay. Those external things "plug in" around it. The
  payoff: we swapped a temporary in-memory store for a real database with **zero changes** to the
  money logic, and the same core serves both the web and the USSD channel.

### 3. The money rules — why this is more than a CRUD app
- **Money is stored as whole numbers** (cents, not dollars-with-a-decimal). Computers are famously bad
  at decimal math (`0.1 + 0.2` isn't exactly `0.3`), and that's unacceptable when it's real money — so
  we never use decimals internally.
- **A double-entry ledger** is the source of truth: every movement is recorded as balanced
  debits and credits, exactly like real accounting. If the books don't balance, the system rejects it.
- **A state machine** governs each payment's life (created → collecting → settling → paid, or →
  refunded / flagged-for-review). Illegal jumps are impossible by design.
- **Idempotency + reconciliation:** retries can never double-charge, and if a confirmation message
  ever goes missing, a "sweep" re-checks the truth with pawaPay. Safety nets, because bugs here move
  real money.

### 4. The payment rails — pawaPay
- **pawaPay** is the third party we rent. Instead of integrating with each mobile-money operator
  ourselves, we call pawaPay's **v2 API** and it handles Vodacom M-Pesa, Airtel, and Orange for us.
- **It's asynchronous:** when we start a payment, pawaPay first says "accepted", then tells us the real
  outcome moments later by calling us back (a **signed callback** — a secure webhook). We verify those
  callbacks cryptographically so we know they genuinely came from pawaPay and weren't tampered with.
- **We are a pure pass-through:** money flows customer → pawaPay → merchant. We never hold it.

### 5. Other ways to pay — the USSD channel
- **USSD** is the `*123#`-style menu that works on any basic phone with no internet. The backend has a
  channel for it so feature-phone users aren't excluded. It calls the *same* money logic as the
  website — written once, reused everywhere. (Wiring it to a real telecom shortcode is a future step.)

### 6. Where the records live — the database
- **PostgreSQL** ("Postgres") — a battle-tested relational database — stores merchants, transactions,
  and the ledger. On Railway it's a managed add-on, so we don't run it ourselves.
- **Alembic** manages the database's structure: when we change the schema, Alembic applies the change
  in a controlled, repeatable way (it runs automatically every time we deploy).
- For automated tests we use **SQLite** (a tiny file-based database) so tests run instantly with
  nothing to set up.

### 7. How it's packaged and hosted
- **Docker** bundles the Python service *and* both web pages into one self-contained image — "it runs
  the same everywhere." One container serves all three (API + console + customer page) from a single
  web address.
- **Railway** hosts that container (the $5/month Hobby plan) and provides the Postgres database and a
  public URL. We picked it because it's cheap and dead-simple. Because it's just a Docker image, we can
  move it elsewhere later with little fuss — the eventual *production* target is **AWS**.
- **A shared password** gates the merchant side of the hosted demo; the customer pages and the pawaPay
  callback stay open (a payer has no login, and pawaPay can't type a password).

### 8. Configuration & secrets
- Settings come from **environment variables** (the "12-factor" approach) — nothing secret is ever
  written into the code or the repository. The pawaPay token and the demo password live only in
  Railway's dashboard; a local `.env` file (never committed) holds them for development.

### 9. How we keep quality high (developer tooling)
- **ruff** — checks code style and catches common mistakes.
- **mypy (strict)** — verifies the code's types line up, catching whole classes of bugs before running.
- **pytest** — the automated test suite (130+ tests). The money rules carry the heaviest testing.
- **Git + GitHub** — version control; Railway rebuilds and redeploys automatically when we push.
- Every change must keep ruff, mypy, and pytest green before it ships.

## At a glance

| Layer | Technology | In one line |
|---|---|---|
| Front-end | HTML / CSS / JavaScript (no framework), segno (QR) | Two simple web pages: merchant console + customer pay |
| Backend language | Python 3.12+ | The money engine and web API |
| Web framework | FastAPI + Uvicorn | Turns Python into the web API the pages call |
| Validation | Pydantic | Guards the shape of every request/response |
| Money core | Double-entry ledger + state machine (custom) | Correctness for real money |
| Payment rails | pawaPay v2 API | Moves the actual mobile money (Vodacom/Airtel/Orange) |
| Extra channel | USSD | Pay from a basic no-internet phone |
| Database | PostgreSQL (+ Alembic; SQLite for tests) | Stores merchants, transactions, the ledger |
| Packaging | Docker | One image with the API + both web pages |
| Hosting | Railway (Postgres add-on) | Runs the container; gives a public URL. AWS is the eventual prod home |
| Config/secrets | Environment variables | No secret ever lives in the code |
| Quality tooling | ruff, mypy --strict, pytest, Git/GitHub | Keeps the code correct and shippable |
